Bulgarian Language & Script Corpus

Datasets & repositories

The BLSC ecosystem is organized as a small set of focused repositories. The core corpus and its metadata live in blsc-core, while this site (blsc-site) hosts public-facing documentation.

Core corpus (blsc-core)

Canonical texts with stable IDs, document-level JSON metadata, genre and period classification, script information, and machine-readable rights.

  • Bulgarian language only (for now).
  • Orthography & period fields tied to historical reforms.
  • Explicit token counts and annotation-layer flags.

Linguistic metadata model

Every document in the corpus is described by a compact JSON object. Fields are intentionally stable, language-agnostic and future-proof, so that additional modules (parallel corpora, annotations) can reference the same identifiers.

  • Document identity: id, author, title, year, source type.
  • Historical dimension: period tied to orthographic reforms, with separate orthography field.
  • Script & language: script (Cyrillic/Latin), dialect (if relevant), normalized-text flag.
  • Annotation layers: booleans for morphological, syntactic and other layers, so consumers know what is available.
  • Rights: explicit license / public-domain marker per document.
Full specification: LINGUISTIC_METADATA_MODEL.md
Example JSON
{"id":"blsc_00001231","author":"Иван Вазов","year":1894,
 "period":"pre_reform_1899","orthography":"pre_reform_1899",
 "genre":"fiction","script":"Cyrillic","token_count":1520,
 "morph_layer":false,"syntax_layer":false,
 "normalized_text":null,"rights":"public_domain"}

The corpus itself is not hard-coded into this site: the JSON metadata lives in the core repository and can be processed with any tooling (Python, R, SQL, etc.).

Annotation guidelines

The BLSC project distinguishes clearly between the corpus (texts + metadata) and optional annotation layers. Annotation guidelines are versioned documents that describe how each layer is produced and how it should be interpreted.

Principles

  • Annotation is always documented and reproducible: every layer has a guideline and a version.
  • Guidelines favour clarity over ideology – they are designed to be used by both linguists and practitioners.
  • Whenever possible, categories are mapped to existing standards (Universal Dependencies, etc.).

Annotation should never silently change the underlying text. Any normalization or correction must be explicitly flagged.

Guidelines document

A living document in the core repository describes the current annotation layers and conventions:

  • Tokenization & sentence segmentation.
  • Morphological features and POS tags.
  • Syntactic relations (if available).
  • Named entities and other semantic labels (future work).

See ANNOTATION_GUIDELINES.md for the current draft.

How to get involved

BLSC is intentionally designed as an open, collaborative effort. If you work with Bulgarian language and would like to contribute, there are several ways to help.

Report issues & ideas

Use the issue tracker in blsc-core for anything related to the corpus, metadata schema, or documentation:

  • Bug reports or inconsistencies.
  • Requests for new metadata fields.
  • Suggestions for additional genres or periods.

Contribute data

If you control rights to suitable Bulgarian texts (public domain, CC-licensed, or with explicit permission), you can propose adding them to the corpus.

  • Provide basic bibliographic information.
  • Clarify licensing and provenance.
  • Open an issue first – do not send large files by email.

Help with annotation

As annotation layers mature, the project may organize focused annotation sprints for specific phenomena (e.g. historical spelling, dialect, named entities).

  • Follow the repository for announcements.
  • Comment on guideline drafts and edge cases.
  • Share evaluation scripts and baselines.

Contact & governance

For questions that do not fit well into GitHub issues (sensitive data, institutional collaborations, etc.) you can reach the maintainers via email. Please keep technical questions on GitHub whenever possible so that answers remain visible to everyone.
